# Подходы к решению: «Скачать на устройство» (краш) и «Удалить фото из альбома» (не срабатывает)

Документ фиксирует всё испробованное и возможные варианты для двух задач. На текущем этапе работы по ним приостановлены.

---

## Задача 1: Удаление фото из альбома — фото не удаляется, список не обновляется

### Симптомы
- Нажатие «Удалить» → «Удаление…» → галерея закрывается, фото в сетке остаётся.
- В логе: запрос `photos.delete` с `owner_id=-1152921504606846756` (подозрительное значение), ответ 200, 14 bytes, в коде «photos.delete ok».

### Уже испробованные подходы (без успеха)

1. **Локальное удаление после успешного API**  
   После `photos.delete` без ошибки: `photos.removeAll { $0.id == photoId }`, `totalCount -= 1`, `fullScreenPhotoItem = nil`. Ожидалось, что список обновится за счёт `@State`. Результат: фото не пропадает.

2. **Защита от перезаписи списка при появлении экрана**  
   В `.onAppear` вызывать `loadPhotos(offset: 0, append: false)` только при `loadState == .idle`, чтобы после закрытия fullscreen не перезатирать уже обновлённый список. Результат: проблема остаётся.

3. **Использование owner_id из объекта фото (для «Сохранённых»)**  
   В модель `VKPhoto` добавлено поле `ownerId`, при формировании `photoIds` и вызове удаления использовалось `photo.ownerId ?? ownerId` альбома. Результат: в запросе уходил некорректный `owner_id` из ответа VK.

4. **Рефетч списка после успешного delete**  
   После успешного `photos.delete`: закрыть галерею и вызвать `loadPhotos(offset: 0, append: false)`, чтобы список приходил с сервера заново. Результат: удаление по-прежнему не происходит (либо API не удаляет, либо ответ не тот).

5. **Возврат к локальному удалению без рефетча**  
   Снова только локальное удаление из `photos` и закрытие fullscreen, без повторной загрузки. Результат: без изменений.

6. **Использование owner_id альбома вместо owner_id из фото**  
   В колбэке удаления в API всегда передаётся `ownerId` альбома (тот, с которым открыт экран), а не `item.ownerId` из фото — чтобы не слать в API некорректное значение (например, вне диапазона Int32). Результат: удаление всё так же не срабатывает.

### Подходы, которые ещё можно попробовать

- **Проверить фактическое тело ответа VK (14 bytes)**  
  Логировать сырой `data`/строку ответа перед декодированием. Убедиться, что приходит именно `{"response":1}`, а не ошибка в другом формате, который мы не парсим как ошибку.

- **Ограничить owner_id диапазоном Int32 при отправке в API**  
  VK использует 32‑битные id. При формировании запроса приводить `owner_id` к допустимому диапазону (например, Int32) или подставлять `ownerId` альбома, если значение из фото «битое».

- **Для альбома «Сохранённые» (-15) вызывать API с текущим пользователем**  
  Явно подставлять в `photos.delete` id текущего пользователя (из auth/profile), а не только owner альбома, если для -15 требуется именно он.

- **Проверить, не нужен ли другой метод VK для «Сохранённых»**  
  Уточнить в документации VK, удаляются ли фото из «Сохранённые» тем же `photos.delete` или другим методом (например, убрать из коллекции без полного удаления).

- **Показать пользователю ошибку при неудаче**  
  При `catch` или при ответе с ошибкой показывать тост/алерт «Не удалось удалить», не закрывать галерею — чтобы было видно, что запрос реально падает.

- **Отладка обновления UI**  
  Логировать `photos.count` и `photoId` до и после `removeAll`; убедиться, что удаляется нужный элемент и что обновляется тот же экземпляр `AlbumPhotosView` (нет ли рассинхрона из‑за замыканий/слоёв навигации).

- **Проверить двойной вызов delete**  
  В логе иногда «photos.delete ok» дважды — убедиться, что кнопка/колбэк не вызываются дважды и что нет гонки между двумя удалениями.

## Задача 2: «Скачать на устройство» — краш при сохранении в «Фото»

### Симптомы
- Нажатие «Скачать на устройство» → краш.
- Сообщение: `Thread N: abort with payload or reason`, очередь: `com.apple.root.default-qos (concurrent)`.
- Адрес в стеке вида `b.lo` (+8) — типично проверка/abort в рантайме или в Photos.

### Уже испробованные подходы (без успеха)

1. **Resume continuation на MainActor**  
   В completion’ах `requestAuthorization` и `performChanges` вызывать `cont.resume(returning:)` внутри `Task { @MainActor in ... }`, чтобы не возобновлять continuation с фоновой очереди. Результат: краш сохраняется.

2. **Вызов performChanges только с main**  
   Загрузку данных и создание `UIImage` вынести в `Task.detached`, вызов `performChanges` обернуть в `MainActor.run { ... }`. Результат: краш сохраняется.

3. **Создание UIImage только внутри блока performChanges**  
   В блок изменений передавать только `Data`; внутри блока делать `UIImage(data: imageData)` и сразу `creationRequestForAsset(from: image)`, чтобы не передавать `UIImage` между потоками. Результат: краш сохраняется.

4. **Сохранение через временный файл**  
   Скачать данные → записать во временный файл → в `performChanges` вызывать только `PHAssetChangeRequest.creationRequestForAssetFromImage(atFileURL: tempFile)` (без `UIImage` в блоке). Результат: краш сохраняется.

5. **Resume через DispatchQueue.main.async**  
   В completion’ах вызывать `DispatchQueue.main.async { cont.resume(returning: ...) }` вместо `Task { @MainActor in ... }`. Результат: краш сохраняется.

6. **Удаление временного файла после completion**  
   Удалять временный файл не в `defer`, а внутри completion после `cont.resume`, чтобы Photos не читал уже удалённый файл. Результат: краш сохраняется.

### Подходы, которые ещё можно попробовать

- **Весь сценарий сохранения выполнять на main**  
  Загрузку по URL, создание `UIImage` и вызов `performChanges` делать строго на главном потоке (например, в одном `MainActor`/`DispatchQueue.main` блоке), без continuation из фоновых completion’ов. Риск: кратковременная блокировка UI.

- **UIImageWriteToSavedPhotosAlbum (deprecated)**  
  Использовать старый API сохранения в «Фото» — другой код внутри системы, возможно без краша на default-qos. Долгосрочно не рекомендуется, но для проверки гипотезы подойдёт.

- **Сохранять в «Файлы» вместо «Фото»**  
  Сохранять изображение в контейнер приложения или в «Файлы» (Documents/Downloads), без вызова PHPhotoLibrary. Пользователь сам переносит в «Фото» при необходимости. Обходит проблему краша, но меняет сценарий.

- **PHPicker / другой API**  
  Проверить, есть ли в текущей версии iOS другой способ «сохранить в Фото» (например, через PHPicker или share-extension), не использующий `performChanges` в том же виде.

- **@Sendable и изоляция блоков**  
  Явно пометить замыкание для `performChanges` как `@Sendable`, убедиться, что в блок не попадают non-Sendable и MainActor-изолированные значения. Исключить захват контекста с главного актора в блок, который выполняется на очереди Photos.

- **Минимальный воспроизводящий пример и репорт в Apple**  
  Собрать минимальный пример: один экран, кнопка «Сохранить», тот же поток (URL → Data → temp file → performChanges(creationRequestForAssetFromImage)). При повторении краша — репорт в Feedback Assistant с версией iOS и стеком.

- **Полный стек краша**  
  В следующей сессии при краше снять полный backtrace падающего потока (все фреймы), чтобы понять: падение в нашем коде, в Swift runtime или внутри Photos.

- **Отключение strict concurrency для модуля/файла**  
  Экспериментально отключить строгую проверку concurrency для файла с сохранением и выполнить весь сценарий через `DispatchQueue.main.async` (без async/await/continuation). Только для локализации причины.

- **Проверка расширения/формата временного файла**  
  Убедиться, что файл имеет корректное расширение (например, .jpg/.png) и что Photos не падает из‑за неизвестного типа. При необходимости конвертировать в JPEG/PNG перед записью.
