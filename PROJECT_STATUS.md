# CleanFeedVK — текущее состояние и план

## 1. Что уже есть (скелет, не MVP)

| Компонент | Статус | Описание |
|-----------|--------|----------|
| **Структура MVVM** | ✅ | Папки: Models, Views, ViewModels, Services (Network, Auth, Persistence), Utilities |
| **Точка входа** | ✅ | `CleanFeedVKApp.swift` + `ContentView.swift` |
| **NetworkService** | ✅ | async/await, URLSession, протокол для DI, проверка HTTP 2xx, декодирование JSON |
| **Диагностика** | ✅ | `AppLogger` (Utilities): уровни debug/info/warn/error, категории, os_log; логирование запросов/ответов в NetworkService |
| **Auth** | ❌ | Папка пустая (OAuth 2.0 + Keychain — по спеке) |
| **Persistence** | ❌ | Папка пустая (SwiftData/кэш — по спеке) |
| **Модели VK** | ❌ | Нет DTO для newsfeed.get, вложений и т.д. |
| **UI лента** | ❌ | Нет списка постов, префетча, фильтров |

Итог: это именно скелет — каркас под MVP, без реального VK API и без ленты.

---

## 2. Куда двигаться (по этапам из спеки)

1. **Skeleton (текущий этап)** — DI-контейнер (опционально), база сети ✅, диагностика ✅.
2. **Auth** — OAuth 2.0 Implicit (WKWebView), Kate Mobile App ID, Keychain для токена.
3. **Data Pipeline** — запрос `newsfeed.get` → парсинг → фильтрация (ads, promo, blacklist) → вывод в консоль.
4. **UI** — подключение пайплайна к SwiftUI (LazyVStack, префетч, ячейка поста).
5. **Offline** — кэш ленты в БД, загрузка из кэша при старте/ошибке сети.

## 3. Что нужно от тебя для подключения VK API

Чтобы перейти к Auth и первому запросу к API, нужны уточнения:

1. **Redirect URI**  
   Для OAuth 2.0 Implicit нужен зарегистрированный redirect (например `https://oauth.vk.com/blank.html` или кастомный). Какой URI указан в настройках приложения VK (или планируешь использовать стандартный blank)?

2. **App ID**  
   В спеке указан Kate Mobile: `2685278`. Подтверждаешь, что используем именно его (без своего Standalone-приложения)?

3. **Секрет**  
   Для чисто клиентского Implicit Flow секрет не обязателен. Нужны ли тебе в будущем серверные вызовы (тогда понадобится Secure Key)?

4. **Тестовый аккаунт**  
   Есть ли отдельный VK-аккаунт для разработки (чтобы не светить личную ленту в логах)?

Ответы на эти пункты достаточно, чтобы заложить Auth (WKWebView + Keychain) и первый вызов `newsfeed.get` с логированием через текущий AppLogger.

## 4. Диагностика — как смотреть, что происходит

- **Консоль Xcode** — в Debug все сообщения с префиксом `[CleanFeedVK]` и категорией (Network, Auth и т.д.) выводятся через `print` и os_log.
- **Console.app** — можно фильтровать по subsystem (bundle id приложения) или по категории.
- **Сеть** — каждый запрос: `→ METHOD URL`; ответ: `← STATUS N bytes`; ошибки и декодирование логируются с уровнем error.

При добавлении Auth и VK-запросов достаточно вызывать `AppLogger.shared.info("Auth", "…")` и т.п. — тогда весь поток будет виден в одном месте.
